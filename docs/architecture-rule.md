# アーキテクチャ設計

> ドメインロジックをフレームワークから独立させることで、テスト容易性を確保し、技術選択の変更コストを下げるため

## このプロジェクトでの判断基準

- 依存の方向は常に外→内
- ドメイン層は外部に依存しない
- テストが書きにくくなったら依存の方向が逆転している兆候
- 「このモジュールを変更する利害関係者は誰か？」で単一責任を判断する
- 関数が長くなったら責務が混入している。Five Lines of Codeを目安に抽出

## 新コードの配置判断

```
ビジネスの本質的なルール → ドメイン
アプリケーション固有の処理フロー → ユースケース
データフォーマットの変換 → アダプター
それ以外 → インフラ/フレームワーク層
```

## モダナイゼーション (Architecture Modernization - Nick Tune)

- **Big Bangリライトは禁止**。Strangler Figパターンで段階的に置換
- なぜ: 全面書き直しはリスクが高く、完成前に要件が陳腐化、段階的置換なら各ステップで価値創出
- 手順: ルーティング層を導入 → 1機能ずつ新システムに移行 → 旧コンポーネントを退役
- 技術だけでなく組織も変える: レガシーを生んだ組織プロセスを変えなければ、新システムも同じ運命を辿る (Conway's Law)
- Wardley Mapで各コンポーネントの進化段階を把握してから移行優先度を決める

## 迷ったら

1. テスト容易性を優先する
2. シンプルな方を選ぶ
3. 変更の影響範囲を最小化する
