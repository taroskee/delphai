# パラメータ設計

> シミュレーションの全数値パラメータに適用する設計原則

## 原則

### 1. 符号なし 0.0〜1.0 統一

全てのシミュレーション数値パラメータは `f32` の `0.0..=1.0` 範囲とする。

- **乗算で範囲保存**: `0.7 * 0.8 = 0.56` → 常に同じ範囲内
- **clampが統一**: 全パラメータに同じ `clamp(0.0, 1.0)` を適用
- **比較が直感的**: 0.0 = 最小/無、1.0 = 最大/完全

### 2. 負の意味は組み合わせから導出

単一パラメータに符号を持たせない。意味はパラメータの**組み合わせ**で表現する。

| awareness | trust | 解釈 |
|---|---|---|
| 0.9 | 0.9 | 熱心な信者 |
| 0.9 | 0.1 | 声の主を認知し反抗 |
| 0.1 | 0.5 | 声の主に無関心 |

| familiarity | trust | 解釈 |
|---|---|---|
| 0.8 | 0.9 | 親友 |
| 0.8 | 0.1 | よく知ってるが敵対 |
| 0.1 | 0.5 | 見知らぬ人、中立 |

### 3. 将来 u8 パック前提

Phase 1 では `f32` のまま実装する。住民数が増加する Phase 2 以降で `u8` (0〜255) へ移行する。

```rust
// 変換（Phase 2+で実装）
fn to_u8(v: f32) -> u8 { (v * 255.0).round() as u8 }
fn from_u8(v: u8) -> f32 { v as f32 / 255.0 }
```

メモリ比較（住民100人 × パラメータ20個）:
- `f32`: 100 × 20 × 4 = 8,000 bytes
- `u8`:  100 × 20 × 1 = 2,000 bytes

全パラメータが同じ 0.0〜1.0 範囲なので変換ロジックは共通1つで済む。

## 現行パラメータ一覧

### Needs（欲求）

| パラメータ | 意味 | 回復手段 | Phase |
|---|---|---|---|
| hunger | 空腹度 | 食事 | 1 |
| fatigue | 疲労度 | 睡眠 | 1 |

### 将来追加予定

| パラメータ | 意味 | 備考 | Phase |
|---|---|---|---|
| thirst | 渇き | 真水で回復。海水は不可（水源の種類が重要） | 2+ |
| nutrition | 栄養状態 | hunger=量、nutrition=質。偏食→病気リスク | 2+ |
| wealth | 相対的な富 | 集団内での相対位置。貨幣経済 | 3+ |

### Needs 拡張パス

Phase 1 では固定フィールド構造体を使用する:

```rust
struct Needs { pub hunger: f32, pub fatigue: f32 }
```

Phase 2+ でフィールド追加が頻繁になった場合、enum インデックス配列に移行する:

```rust
#[repr(u8)]
enum NeedKind { Hunger = 0, Fatigue = 1, Thirst = 2, Nutrition = 3 }

struct Needs { values: [f32; NeedKind::COUNT] }

impl Needs {
    pub fn hunger(&self) -> f32 { self.values[NeedKind::Hunger as usize] }
    // ... accessor methods でAPI互換を保つ
}
```

### Relationship（関係性）

| パラメータ | 意味 |
|---|---|
| familiarity | 相手の認知度。0.0=見知らぬ人、1.0=親密 |
| trust | 相手への信頼。0.0=不信、1.0=完全な信頼 |

### Citizen（市民固有）

| パラメータ | 意味 |
|---|---|
| divine_awareness | プレイヤーの声への認知度。0.0=無認知、1.0=完全認知 |
