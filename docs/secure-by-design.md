# Secure by Design ガイドライン

> 不正な状態を「実行時」ではなく「設計・型レベル」で排除する
> すべての入力・境界・権限は「信頼しない前提」で扱う

---

## 基本原則

- 不正な状態は型レベルで表現できないようにする
- 実行時チェックよりコンパイル時保証を優先する
- Fail Safe: 不明・不正な入力は拒否する
- Fail Closed: エラー時は安全側に倒す (デフォルト拒否)
- 各コンポーネントは必要最小限の権限のみを持つ
- 権限は明示的に付与し、暗黙に拡張しない
- システム内の「信頼できる領域」と「外部入力」を区別する
- 意味を持つ値はドメイン型 (値オブジェクト)で表現する
- 外部入力はすべて境界で検証後、ドメイン型に変換する

## データと型の設計

- ドメイン層では「常に正しい値」が来る前提にする
- データは原則、不変型とし、状態変更は新しい値として生成する
- null / undefined の乱用を避ける
- 外部とのやり取りはDTOで行う
- ドメインモデルを直接外部に公開しない

## シークレット、認証、メッセージ

- シークレットはコードに埋め込まず、環境変数またはシークレットマネージャを使用する
- シークレットをログに出力しない
- 認可は明示的に行う (暗黙の許可をしない)
- 認可は必ずサーバー側で検証する
- クライアントの状態を信用しない
- エラーメッセージに内部情報を含めない
- スタックトレースは外部に出さない
- ユーザーエラー (4xx)、システムエラー (5xx)を明確に分離する

## 依存関係とサプライチェーン

- 必要なパッケージのみを導入する
- 使っていない依存は削除する
- 脆弱性スキャンをCIで自動実行する
- メンテナンスされているライブラリのみ採用する

## CLAUDE.md のセキュリティ

- CLAUDE.mdの変更は通常コード以上のレビューを必須とする
- 少なくとも1つ以上の信頼境界を明示する
- 設計変更がセキュリティ制約を破らないか確認する

## レビュー時チェックリスト

PRレビュー時は以下を確認する:

- [ ] 外部入力はすべて検証されているか
- [ ] ドメイン型が適切に使われているか
- [ ] 不変性が保たれているか
- [ ] シークレットが安全に扱われているか
- [ ] ログに機密情報が含まれていないか
- [ ] 権限が過剰になっていないか
- [ ] 依存パッケージは最小限か
- [ ] エラー時に安全側に倒れるか
