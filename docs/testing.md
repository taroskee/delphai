# テスト戦略

> 検証ループが品質を2〜3倍にする(Boris Cherny実証)
> テストなき修正は技術的負債の入口であり、CIの信頼性を破壊する

## このプロジェクトでの判断基準

- 不具合修正は**必ずテストで再現してから**修正、例外なし
- テストが書きにくい場合は設計が悪い兆候、設計を見直せ
- モックは外部境界(プロセス外依存)のみ、ドメインロジックのモックは設計の問題

## Unit Testing PPP(Khorikov, Manning)

- **良いテストの4本柱**: 1.退行からの保護、2.リファクタリング耐性、3.迅速なフィードバック、4.保守容易性、最初の2つは妥協するな
- **振る舞いを検証し、実装を検証するな**: テストが実装詳細に結合するとリファクタリングで壊れるので、観察可能な振る舞い(戻り値、状態変化、外部呼び出し)だけを検証
- **テストの価値 = 保護 × 耐性 / コスト**: 4本柱のどれかがゼロなら価値もゼロ、カバレッジ率を目標にするな
- **モックの使い所**: プロセス外の管理できない依存(外部API、メール送信等)のみ、DBなどプロセス外でも管理できる依存は実物を使え
- **テストも剪定する**: 価値のないテストは削除、テストが多い≠品質が高い

## CI/CD

- TDD/CI/CDは基本前提、スキップする場合は明示的に宣言する
- CI失敗は指示を待たず自律的に修正する
- PostToolUseフックでlint/format自動化し、CI失敗を未然に防ぐ
- パフォーマンス回帰を防ぐためベンチマークをCIに組み込む

## CLAUDE.md自体のeval

ルール変更の効果はBuild/Lint/Test通過率とミス率で測定し、効果のないルールは消す
